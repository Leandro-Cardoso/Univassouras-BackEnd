Git do Zero

Nesta aula vamos aprender a instalar e configurar o Git do zero, desde verificar se está instalado, até criar um repositório local e conectá-lo ao GitHub.

1. Testando se o Git já está instalado

Abra o Prompt de Comando (cmd) e digite:

git --version
  • Se aparecer a versão do Git (ex: git version 2.45.0), já está instalado.
  • Se aparecer 'git' não é reconhecido, significa que você ainda não tem o Git instalado.
🔎 Para leigos: é como perguntar para o computador “você já sabe falar Git?”. Se ele responder com uma versão, então está tudo certo.

2. Baixando e Instalando o Git

  1. Acesse o site oficial: https://git-scm.com/downloads
  2. Escolha a versão para seu sistema operacional (Windows, Mac ou Linux).
  3. No Windows: execute o instalador e deixe as opções padrão.
  4. Finalize a instalação e abra novamente o cmd.
  5. Digite novamente: 
    git --version
    para confirmar que foi instalado corretamente.
💡 Dica: O Git instala também o Git Bash, um terminal mais completo e usado por desenvolvedores.

3. Configurando o Git pela primeira vez

No terminal, defina seu nome e email (eles aparecerão nos seus commits):

git config --global user.name "Seu Nome"
git config --global user.email "seuemail@example.com"

Para confirmar suas configurações:

git config --list
👶 Para leigos: é como assinar suas tarefas com nome e e-mail. Assim, quando você “salva uma versão” (commit), todos sabem quem foi o autor.

🔑 Removendo credenciais antigas

Se houver outro usuário cadastrado e você quiser trocar:

  1. No Windows, abra o Painel de Controle → Gerenciador de Credenciais.
  2. Vá em Credenciais do Windows.
  3. Procure por entradas relacionadas ao Git, GitHub, GitLab ou Bitbucket, ex: git:https://github.com.
  4. Remova essas entradas.
  5. Na próxima vez que fizer git pull ou git push, o Git pedirá novo usuário e senha/token.
🔄 Para leigos: é como apagar credenciais antigas do seu caderno, para poder escrever com seu novo nome.

4. Criando um Repositório Local

  1. Escolha uma pasta no seu computador, ou crie uma nova: 
    mkdir MeuProjeto
cd MeuProjeto
  2. Inicialize o repositório Git dentro da pasta: 
    git init
    Isso cria uma pasta oculta .git que guarda todo o histórico do seu projeto.
  3. Crie um arquivo para teste, ex: 
    echo "# Meu Primeiro Projeto Git" > README.md
  4. Adicione e faça o commit inicial: 
    git add .
git commit -m "Primeiro commit"
📂 Para leigos: o repositório local é como uma pasta mágica no seu computador que lembra de tudo que você faz. Cada commit é como uma foto tirada do estado atual da pasta.

5. Criando um Repositório no GitHub

  1. Acesse GitHub e crie sua conta.
  2. No canto superior direito, clique em New Repository.
  3. Defina um nome, por exemplo: meu-projeto.
  4. Não precisa marcar a opção de README (já criamos localmente).
  5. Clique em Create repository.
☁️ Para leigos: o GitHub é como o Google Drive dos programadores, mas com histórico completo das mudanças.

6. Conectando o Repositório Local ao GitHub

git branch //verifique o branch criado (master ou main)
git remote add origin https://github.com/seu-usuario/meu-projeto.git
git branch -M branch-criado
git push -u origin branch-criado
🚀 Para leigos: é como dizer “a partir de agora, essa pasta local está ligada à nuvem”. E com o push você faz o upload das suas fotos (commits).

7. Fluxo Básico do Dia a Dia

  1. Editar ou criar arquivos no projeto.
  2. Adicionar mudanças: 
    git add .
  3. Salvar as alterações no histórico: 
    git commit -m "Mensagem do que foi feito"
  4. Enviar para o GitHub: 
    git push
🔄 Para leigos: pense no ciclo como “editar → preparar → fotografar → enviar para a nuvem”.

📖 Paralelo com Git

  • 📂 Diretório Local → Onde você trabalha nos arquivos.
  • 📝 Staging Area → Uma espécie de “sala de espera” antes de registrar no histórico.
  • 📜 Repositório Local → Onde ficam seus commits no computador.
  • ☁️ Repositório Remoto → A nuvem (GitHub, GitLab, Bitbucket), onde você compartilha com os outros.

8. O que é o .gitignore?

O arquivo .gitignore diz ao Git quais arquivos ou pastas devem ser ignorados, ou seja, não precisam ser versionados.

🗑️ Para leigos: imagine que você tem uma mochila para levar coisas para a escola. Você só quer levar livros e cadernos (código fonte), mas não quer levar lixo ou papéis amassados (arquivos temporários). O .gitignore é a lista que diz o que não deve ir para a mochila.

Exemplo de .gitignore para uma solução .NET Core:

# Arquivos de usuário do Visual Studio
*.suo
*.user
*.userosscache
*.sln.docstates

# Build results
[Bb]in/
[Oo]bj/
[Ll]og/
[Dd]ebug/
[Rr]elease/
x64/
x86/
[Aa][Rr][Mm]/
[Aa][Rr][Mm]64/

# Arquivos temporários
*.tmp
*.temp
*.log
*.vspscc
*.vssscc
*.cache
*.pdb

# Diretórios do Visual Studio
.vs/
TestResults/
.nuget/
packages/

# Arquivos de configuração de publicação
*.pubxml
*.publishsettings

# Arquivos de backup
*.bak
*.backup
*.old

# Arquivos do Resharper / Rider
_ReSharper*/
*.DotSettings.user
.idea/

# Arquivos de sistema
Thumbs.db
Desktop.ini

  • Instalamos o Git pelo site oficial.
  • Configuramos nome e email.
  • Criamos um repositório local.
  • Subimos o projeto para o GitHub.
  • Aprendemos sobre o .gitignore no contexto .NET Core.
  • Aprendemos a remover credenciais antigas no Windows.

🔑 Resumindo para leigos: O Git é como um histórico escolar do seu projeto. Cada commit é uma nota salva no boletim. O GitHub é o cartório onde você registra oficialmente seu histórico e compartilha com o mundo. O .gitignore é como a regra que diz: “isso aqui não precisa ir para o cartório”.

✅ Agora você já sabe instalar, configurar, usar o Git do zero e gerenciar credenciais antigas. Pratique criando e subindo seus próprios repositórios!
